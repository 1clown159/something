## 1. 项目概述与选题背景

### 1.1 选题背景

在地震勘探、气象模拟、科学计算等领域，海量浮点数据（float32）的存储和传输开销巨大。以地震数据为例，单次三维地震勘探可产生数亿乃至数十亿个采样点，原始数据量可达 GB 甚至 TB 级别。传统的通用压缩算法（如 zstd、lzma）虽然能提供一定的压缩比，但未充分利用浮点数据的结构特征和空间相关性。

本课题研究了一种基于深度学习的无损浮点数据压缩算法——**Stage4**，针对 float32 格式数据，利用卷积神经网络（CNN）对指数位进行概率预测，结合算术范围编码（Range Coding）实现高效无损压缩，并构建了完整的可视化演示与分析平台。

### 1.2 项目目标

- 实现 Stage4 无损浮点压缩算法，包括模型训练、推理、编码/解码全流程
- 构建 Web 可视化平台，提供交互式算法演示和数据压缩工具
- 支持 SEG-Y 地震数据格式的解析、可视化和压缩
- 提供通用压缩算法对比分析

### 1.3 项目规模

| 维度 | 数据 |
|------|------|
| 算法核心代码 | ~5000 行 Python |
| 后端 API | ~1500 行 Python（FastAPI） |

| CSS 样式 | 8 个模块化 CSS 文件 |

| 总代码量 | 约 10000+ 行 |

## 2. 核心算法：Stage4 无损浮点压缩

### 2.1 IEEE 754 浮点数结构与位拆解

float32 类型在计算机中按 IEEE 754 标准存储，共 32 位：

```
31  30      23  22                      0
┌────┬─────────┬──────────────────────────┐
│Sign│ Exponent │        Mantissa          │
│ 1b │   8b     │         23b              │
└────┴─────────┴──────────────────────────┘

value = (-1)^sign × 2^(exponent-127) × (1 + mantissa/2^23)
```

**三部分位拆解方法**（核心代码在 `algorithm/common.py:179-184`）：

```python
def extract_float_components(data_float32):
    data_u32 = np.asarray(data_float32, dtype=np.float32).view(np.uint32)
    signs   = ((data_u32 >> 31) & 0x1).astype(np.uint8)      # 符号位：1 bit
    exps    = ((data_u32 >> 23) & 0xFF).astype(np.uint8)     # 指数位：8 bits
    mants   = (data_u32 & 0x7FFFFF).astype(np.uint32)        # 尾数位：23 bits
    return signs, exps, mants
```

利用 NumPy 的 `view` 原地重解释位模式，无需逐点循环，效率极高。

### 2.2 算法核心思想

Stage4 的核心思路是**非对称压缩策略**——只对 float32 的 8 位指数部分使用深度学习模型进行概率预测和熵编码，而对符号位和尾数位采用经典压缩方法处理。

**整体流程**：

```
原始 float32 数据
        │
        ▼
  extract_float_exponents()     ← 提取 8-bit 指数（体积压缩到 1/4）
        │
        ▼
  build_stage4_features()       ← 构建因果特征图（对角线扫描顺序）
        │
        ▼
  Small2DCNN.forward()          ← 输出 256 类概率分布
        │
        ▼
  softmax → CDF                 ← 量化为范围编码的累积分布函数
        │
        ▼
  RangeEncoder.encode_symbols() ← 按 CDF 编码真实符号，生成位流
        │
        ▼
  .s4rc bitstream               ← 压缩后的指数位流
  + sign/mantissa 辅助压缩      ← zlib/zstd/lzma 压缩
        │
        ▼
  总压缩结果                     ← 指数 + 符号 + 尾数压缩数据
```

### 2.3 LOCO-I 预测器与残差模式

**LOCO-I 预测器**（`algorithm/stage4.py:84-94`）：

```
pred = clamp(left + up - up_left, 0, 255)
```

边界情况：
- `(t<=0, s<=0)` 返回 0
- `t<=0` 返回左邻 `plane[t, s-1]`
- `s<=0` 返回上邻 `plane[t-1, s]`

**残差模式**（推荐）：

```
target_symbol = (真实指数 - LOCO-I预测值) mod 256
```

解码时：

```
真实指数 = (LOCO-I预测值 + symbol) mod 256
```

残差模式使得编码目标集中在 0 附近，大幅降低了信息熵，显著提高压缩比。

### 2.4 因果特征图构建

为保证无损解码的正确性，特征构建必须遵循**因果约束**：特征图的每个像素只能使用当前点之前（因果关系上可见）的信息。

**对角因果扫描顺序**（`diagonal_causal_edge` 模式）：

```
可见条件：(tt + ss) < (t + s)
即：对角线索引之和小于当前点的所有像素为可见
```

**6 通道特征图**（`residual` 目标模式）：

| 通道 | 名称 | 值域 | 含义 |
|------|------|------|------|
| 0 | 像素值 | 0-255（÷255 归一化） | 因果可见区域的实际指数值 |
| 1 | 可用掩码 | {0, 1} | 真实或映射的可用像素 |
| 2 | 因果掩码 | {0, 1} | 对角因果顺序下可见的像素 |
| 3 | 映射掩码 | {0, 1} | 从邻居"克隆"过来的像素（边界外推） |
| 4 | 预测值 | 常量 | LOCO-I 预测值（全图统一） |
| 5 | 残差值 | 0-255（÷255 归一化） | (实际值 - 预测值) mod 256 |

**4 通道特征图**（`raw` 目标模式）：通道 0-3，不含预测值和残差。

### 2.5 神经网络模型：Small2DCNN

模型结构（`algorithm/stage4.py:64-81`）：

```
Small2DCNN:
  Conv2d(in_channels, base_channels=16, 3x3, padding=1) → ReLU
  Conv2d(16, 32, 3x3, padding=1) → ReLU
  Conv2d(32, 32, 3x3, padding=1) → ReLU
  AdaptiveAvgPool2d(1)           → 全局平均池化
  Flatten → Linear(32, 256)      → 256 类 logits 输出
```

设计特点：
- **轻量化结构**：仅 3 层卷积 + 1 个全连接层，参数量约 2 万，适合实时推理
- **全局平均池化**：替代传统全连接层前的 Flatten，减少参数量且增强泛化能力
- **256 类输出**：对应 8-bit 指数所有可能取值（0-255）
- **输入通道自适应**：根据特征模式和目标模式动态调整输入通道数（4 或 6）

**训练配置**：
- 优化器：AdamW（lr=1e-3, weight_decay=1e-4）
- 损失函数：CrossEntropyLoss
- 训练样本：50万（train） + 5万（val）
- 训练轮次：120 epochs
- Batch size：128
- Patch 尺寸：9×17（trace × sample）

### 2.6 算术范围编码（Range Coding）

范围编码器是熵编码的核心组件（`algorithm/range_coder.py`），采用 32 位状态精度。

**编码过程**（简化）：

```
输入: CDF[257], symbol
流程:
  range = high - low + 1
  low   = low + range * CDF[symbol] / total
  high  = low + range * CDF[symbol+1] / total - 1
  归一化：当 low 和 high 的高位相同时，输出确定位
重复直到所有符号编码完成 → 最终区间内任意整数 = 编码结果
```

**关键实现特点**：
- 支持 Numba JIT 加速（`@njit(cache=True)`），大幅提升编码/解码吞吐
- CDF 由 softmax 概率经整数化量化得到（`codec.py:65-91`）
- 确保 CDF 总和为 2^15（32768），总范围精确匹配

### 2.7 编码布局：Global Diagonal

推荐使用 `global_diag` 布局（`Stage4GlobalDiagonalRangeCodec`）：

- 按对角线坐标（p + t + s）排序所有体素点
- 逐对角线批量推理模型 → 获得所有点的概率分布
- 批量编码符号，最大化 GPU 利用率
- 比逐点 `raster` 编码快数个数量级

### 2.8 符号位和尾数位的压缩

符号位和尾数位不经过 Stage4 模型，而使用多种经典压缩器处理：

| 方法 | 适用场景 | 特点 |
|------|---------|------|
| Bitpack | 随机/均匀数据 | 直接紧凑打包，无压缩但无开销 |
| RLE | 连续重复数据 | 适合符号位或尾数位连续相同的情况 |
| zstd | 通用 | 现代 LZ77 变体 + FSE 熵编码，综合性能好 |
| lzma | 高压缩比需求 | LZMA2 算法，高压缩比但速度慢 |
| PathL-lite | 实验性 | 针对性优化的轻量级编码 |

在 `compute_aux_bits.py` 中自动比较各方案的压缩后尺寸，选择最优方案。

## 5. 关键技术细节

### 5.1 SEG-Y 格式解析

SEG-Y 是地震勘探数据的标准存储格式：

```
┌──────────┬──────────┬─────────────────────────────────┐
│ 3200 B   │  400 B   │  Trace 0  │ Trace 1 │ ...      │
│ 文本头   │ 二进制头 │ 240B+数据  │ 240B+数据│          │
│ (EBCDIC) │ (参数)   │            │         │          │
└──────────┴──────────┴─────────────────────────────────┘
```

**二进制头关键字段**：

| 字节位置 | 字段 | 说明 |
|---------|------|------|
| 16-18 | sample_interval | 采样间隔（微秒） |
| 20-22 | sample_count | 每道采样点数 |
| 24-26 | format_code | 1=IBM浮点, 5=IEEE浮点 |

**道头关键字段**（每道 240 字节）：

| 字节位置 | 字段 | 说明 |
|---------|------|------|
| 72-80 | SourceX, SourceY | 震源坐标 |
| 188-196 | INLINE_3D, CROSSLINE_3D | 三维测线索引 |

**IBM 浮点格式转换**：

```
IBM float (32-bit):
value = (-1)^sign × 16^(exp - 64) × mantissa / 2^24

与 IEEE 754 的关键区别：
- IEEE 754 基数为 2，IBM 基数为 16
- IBM 浮点表示范围更大但精度略低
```

## 6. 开发过程中的困难与解决方案

### 6.1 模拟压缩问题

**问题描述**：Web 可视化平台的压缩功能最初使用了模拟压缩——将数据文件的前半部分直接截断返回，导致所有文件的"压缩比"固定为 2:1，并非真实的 Stage4 算法压缩。

**根本原因**：
- 代码中 `stage4_bridge.py` 仅做了 `data[:len(data)//2]` 的简单截断
- 未实际调用 `Stage4GlobalDiagonalRangeCodec` 进行编码

**解决方案**：
1. 替换为真实调用 `Stage4GlobalDiagonalRangeCodec.encode_exponents()` 执行压缩
2. 引入 `codec.py` 和 `range_coder.py` 的正确导入路径
3. 添加 WebSocket 实时进度推送，显示对角线编码进度
4. 增加 SHA256 校验确保解压数据与原始数据逐位一致

**相关文件**：`docs/FIXES.md:1-135`

### 6.2 训练配置与推理配置不匹配

**问题描述**：模型推理时报错：
```
RuntimeError: Given groups=1, weight of size [8, 6, 3, 3],
expected input[1, 4, 9, 17] to have 6 channels, but got 4 channels instead
```

**根本原因**：模型检查点使用 `target_mode=residual`（6 通道输入）训练，但配置默认使用 `target_mode=raw`（4 通道输入）。

**解决方案**：
1. 实现自动从检查点读取配置：`checkpoint.get('feature_mode')` / `checkpoint.get('target_mode')`
2. 前端默认配置同步更新为 `residual` 模式
3. 添加配置校验逻辑，推理前检查输入通道数是否匹配

### 6.3 数据形状推断困难

**问题描述**：用户上传的 .bin/.dat 文件不包含维度信息，需要推断三维形状 `(profiles, traces, samples)`。

**解决方案**（四级回退策略，见 5.2 节）：
1. 查找 `.shape` Sidecar 文件
2. 已知测试文件形状表匹配
3. 通用推断函数（基于文件字节数分解）
4. 启发式回退（尝试常见采样数）

### 6.4 SVG 渲染性能瓶颈

**问题描述**：SEG-Y 热力图最初使用 D3.js SVG 渲染，当剖面尺寸为 600×2001（120 万像素点）时，创建 120 万 DOM rect 元素耗时 5-10 秒，浏览器卡死。

**解决方案**：
1. 将热力图渲染从 SVG 切换为 Canvas 2D
2. 使用 `CanvasRenderingContext2D.createImageData()` 直接操作像素缓冲区
3. 采用蓝-白-红色谱映射：`(振幅_min → 蓝, 0 → 白, 振幅_max → 红)`
4. 渲染时间从 5-10s 降至 <100ms

### 6.5 SEG-Y 解析库依赖问题

**问题描述**：标准 Python SEG-Y 解析库 `segyio` 依赖 C 编译环境（需要 C 编译器 + SDK），Windows 环境安装困难。

**解决方案**：
1. 自研纯 Python SEG-Y 解析器（`sgy_parser.py`，~130 行）
2. 实现功能：
   - EBCDIC → ASCII 文本头解码
   - 二进制头解析（`struct.unpack`）
   - 道头扫描与 inline/crossline 坐标收集
   - IBM float ↔ IEEE float 格式自动转换
   - 网格维度启发式推断
3. 支持大端序和小端序文件
4. 零外部依赖，跨平台兼容

### 6.6 大量数据逐点循环效率低

**问题描述**：对 4.2 亿采样点的 SEG-Y 文件进行位统计，使用 Python 逐点循环，预估耗时 10+ 分钟。

**解决方案**：
1. 将 `for s in range(samples)` 循环替换为 `numpy.frombuffer → reshape(-1, 4)` 向量操作
2. 使用 `np.bincount()` 替代 Python 累加计数
3. 使用 `np.count_nonzero()` 替代逐位判断
4. 添加内存缓存（`_bit_stats_cache`、`_sgy_meta_cache`）避免重复扫描
5. 最终耗时从 10+ 分钟降至约 50 秒

### 6.7 项目模块化重构

**问题描述**：开发初期项目目录混乱，多个子项目（`20260420/`、`experiments/` 等）混放在根目录，代码中充斥硬编码绝对路径 `C:\Users\32599\Desktop\thesis\...`。

**解决方案**：
1. 重新组织为四大模块：`algorithm/`、`visualizer/`、`scripts/`、`docs/`
2. 将所有硬编码路径替换为 `Path(__file__).resolve().parent` 动态推导
3. 创建 `__init__.py` 使各模块可相互导入
4. 统一导入策略：`try/except ImportError` 同时支持包内导入和直接脚本运行

### 6.8 前端页面拆分与状态管理

**问题描述**：最初为单一 `index.html` 页面，4 个功能步骤挤在一起，代码维护困难。

**解决方案**：
1. 拆分为 5 个独立 HTML 页面，每个页面专注一个功能模块
2. 提取通用组件：
   - `StepNavigation` 类：统一步骤导航逻辑
   - `AppState` 对象：全局状态管理
   - `api.js`：统一 HTTP 通信层
   - `utils.js`：通用工具函数
3. CSS 拆分为 8 个模块化文件（variables / layout / components / 各页面专用样式）

## 7. 实验结果与性能分析

### 7.1 压缩性能（测试数据）

| 数据类型 | 原始大小 | 压缩后 | 压缩比 | 码率 | 分析 |
|---------|----------|--------|--------|------|------|
| 常量数据 | 156.25 KB | 3.15 KB | **49.63x** | 0.64 bits/voxel | 所有值相同，CNN 完美预测 |
| 渐变数据 | 156.25 KB | 3.12 KB | **50.03x** | 0.64 bits/voxel | LOCO-I 对线性数据极佳 |
| 地震模拟 | 156.25 KB | 38.42 KB | **4.07x** | 7.87 bits/voxel | 有统计规律但含噪声 |
| 随机数据 | 156.25 KB | 41.72 KB | **3.74x** | 8.55 bits/voxel | 随机数据仍不膨胀 |

### 7.2 压缩性能分析

1. **常量/渐变数据**：压缩比接近 50x，LOCO-I 预测器 + 残差模式完美工作，预测概率分布呈现单一尖峰
2. **类地震数据**：压缩比约 4x，码率 7.87 bits/voxel，相比原始 8 bits 指数有所改善但噪声限制了进一步压缩
3. **随机数据**：压缩比约 3.74x，码率 8.55 bits/voxel，即使无法预测，算法也不会膨胀数据
4. **总体指标计算**：总字节数 = 指数位压缩字节 + 符号位压缩字节 + 尾数字节

### 7.3 无损验证

所有压缩测试均通过 SHA256 校验，确认压缩-解压后数据与原始数据逐 bit 完全一致，实现真正的无损压缩。

## 8. 技术创新点总结

### 8.1 算法创新

1. **非对称压缩策略**：仅对 8 位指数使用深度学习熵编码，符号/尾数使用经典方法，在计算效率和压缩比之间取得平衡
2. **对角因果特征图**：通过对角线扫描顺序实现因果约束下的最大上下文信息利用
3. **LOCO-I + 残差预测**：将经典的 LOCO-I 线性预测器与神经网络结合，通过残差模运算显著降低编码熵
4. **多通道特征表示**：6 通道特征图同时编码像素值、因果掩码、映射掩码、预测值和残差，为模型提供丰富的上下文信息

### 8.2 工程创新

1. **三级编码布局**：raster（逐点）→ tile64（分块）→ global_diag（全局对角线批量），灵活平衡速度与内存
2. **Numba JIT 范围编码器**：将熵编码的 Python 实现通过 Numba 编译为机器码，速度提升 10 倍以上
3. **纯 Python SEG-Y 解析器**：零依赖实现 EBCDIC 解码、IBM/IEEE 浮点转换、维度推断
4. **Canvas 替代 SVG 渲染**：百万级数据点可视化性能提升 100 倍

### 8.3 系统创新

1. **全链路可视化**：从位级拆解到概率预测到熵编码，每一步都可交互查看
2. **多算法对比平台**：支持 Stage4、zstd、lzma、RLE、bitpack 等方法的横向比较
3. **WebSocket 实时进度**：对角线编码进度实时推送，避免长任务"假死"的用户体验问题
