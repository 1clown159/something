## 1. 功能概述

灰度图局部分析是 SEG-Y 查看器的扩展模块，允许用户在灰度地震剖面上框选一个局部区域（30 道 × 40 样点），实时获取该区域内的**位级别统计信息**，并将其渲染为四个可视化图表。

**设计目标**：从"宏观地震振幅浏览"下沉到"微观位结构分析"，揭示 float32 数据在 bit 层面的统计特性，验证 Stage4 算法"分离压缩指数位"这一核心策略的数据依据。

**核心洞察链路**：

```
灰度图框选局部区域
       ↓
     位分解 (sign / exponent / mantissa)
       ↓
     四个渲染图展示位分布
       ↓
     指数位图上的纵线拖拽 → 振幅曲线
       ↓
     验证: 指数位具有空间相关性 → CNN可建模 → 压缩有效
```

## 3. 灰度图像渲染

### 3.1 渲染原理

灰度图使用 Canvas 2D `createImageData` + `putImageData` 进行像素级渲染：

```javascript
// 伪代码
for (trace = 0; trace < numTraces; trace++)
    for (sample = 0; sample < numSamples; sample++)
        灰度值 = (amplitude[trace][sample] - vMin) / (vMax - vMin) * 255
        pixel[trace][sample] = RGB(灰度值, 灰度值, 灰度值)
```

每个数据点占 2×1 像素（横向 2px = 道的宽度，纵向 1px = 样点的高度），整幅图像使用 `image-rendering: pixelated` 保持像素锐利。

### 3.2 与热力图的区别

| 特性 | 热力图 | 灰度图 |
|------|--------|--------|
| 色阶 | 蓝→白→红（发散型） | 黑→白（0-255 灰度） |
| 鼠标交互 | 悬停显示数据详情 tooltip | **观测框**跟随鼠标 |
| 点击交互 | （已禁用） | **固定观测框**触发位统计 |
| 用途 | 快速浏览整体振幅分布 | 精确定位局部区域进行位分析 |

### 3.3 坐标映射

灰度图使用**实际显示的 CSS 像素**计算鼠标位置与观测框坐标：

```javascript
function getCSSPixelSize() {
    var rect = canvas.getBoundingClientRect();
    return {
        cssW: rect.width / numTraces,   // 每道的实际显示像素宽度
        cssH: rect.height / numSamples, // 每样点的实际显示像素高度
    };
}
```

这确保了无论 canvas 如何缩放（`max-width: 100%`），观测框始终与图像内容对齐。

## 4. 观测框机制

### 4.1 设计

观测框是一个绝对定位的 `<div>` 元素，覆盖在灰度图 canvas 上方。它有两种状态：

| 状态 | 颜色 | 交互 |
|------|------|------|
| **跟随**（未固定） | 橙色 `#f59e0b` 虚线 | 跟随鼠标移动，自动隐藏 |
| **固定**（已点击） | 粉色 `#ec4899` 虚线 | 锁定位置，触发位统计，再次点击解除 |

### 4.2 坐标计算

观测框始终以鼠标位置为中心：

```javascript
// 框左上角坐标 = 鼠标坐标 - 框的一半尺寸（边界裁剪）
boxLeft = Math.max(0, Math.min(mouseX - boxWidth/2, numTraces - boxWidth))
boxTop  = Math.max(0, Math.min(mouseY - boxHeight/2, numSamples - boxHeight))
```

框的大小为 **30 道 × 40 样点**，在典型剖面（300 道 × 2500 样点）中覆盖约 10% 的宽度和 1.6% 的高度。

### 4.3 容器偏移补偿

由于 canvas 在容器内可能居中（`margin: 0 auto`），观测框定位需加上 canvas 在容器内的偏移：

```javascript
obsOverlay.style.left = (canvas.offsetLeft + boxLeft * cssW) + 'px';
obsOverlay.style.top  = (canvas.offsetTop  + boxTop  * cssH) + 'px';
```

## 5. 四个渲染图详解

### 5.1 局部放大（Raw Amplitude）

**数据源**：`raw_grid` — 选中区域的原始 float32 振幅值。

**渲染**：灰度图（0-255），使用 **2% percentile clip** 去除极端离群值：

```javascript
// 裁剪到 [2%, 98%] 分位数之间
lo = vMin + 0.02 * (vMax - vMin);
hi = vMax - 0.02 * (vMax - vMin);
gray = clamp((value - lo) / (hi - lo), 0, 1) * 255;
```

**转置渲染**：数据存储为 `grid[trace][sample]`，渲染时转置为「道=X轴, 样点=Y轴」，与灰度图方向一致。

**配色**：仿 matplotlib `cmap="gray"`。

**意义**：直接展示选中区域的振幅分布，与灰度图同一位置形成对照。

---

### 5.2 符号位分布（Sign Field）

**数据源**：`sign.grid` — 每个采样点的符号位（0 = 正数 / 零, 1 = 负数）。

**渲染**：二值像素图，使用离散色映射：

| 值 | 含义 | 颜色 |
|----|------|------|
| 0 | 正数 / 零 | `#F4F4F2` 米白 |
| 1 | 负数 | `#0B79B4` 深蓝 |

**右上角图例**：白色圆角框 + 色块 + 标签 `0` / `1`。

**意义**：符号位是 float32 中**最纯随机的成分**——每个 bit 约 50% 概率为 0 或 1，且相邻采样点无空间相关性。这解释了为什么 Stage4 **不对符号位做 CNN 建模**，而是用 zlib 直接压缩。

**数据分析洞察**：在真实地震数据中，符号位确实呈现接近均匀的二值分布。蓝色（1=负）和米白（0=正）的像素大致各占一半，无明显的块状结构——验证了"符号位不可预测"的假设。

---

### 5.3 指数位分布（Exponent Field）

**数据源**：`exponent.grid` — 每个采样点的 8-bit 指数值（范围 0-255）。

**渲染**：使用 **Viridis 色阶**（深紫→青→绿→黄），连续映射 0-255 到色阶。

**Viridis 色阶**（JavaScript 实现）：
```javascript
function viridis(t) {
    // 0.00 → dark purple (68, 1, 84)
    // 0.25 → teal (69, 128, 173)
    // 0.50 → green (69, 191, 138)
    // 0.75 → yellow-green (190, 223, 83)
    // 1.00 → bright yellow (253, 231, 37)
}
```

**三条可拖拽纵线**：

| 线 | 颜色 | 十六进制 | 说明 |
|----|------|---------|------|
| L1 | 🔴 红色 | `#E53935` | 初始在 25% 道位置 |
| L2 | 🔵 亮蓝 | `#00B7FF` | 初始在 50% 道位置 |
| L3 | 🟠 橙色 | `#FF8F00` | 初始在 75% 道位置 |

**交互**：鼠标悬停纵线变为 `col-resize` 光标，按下拖动左右移动。线为**虚线**（`[5, 3]` dash 模式），线宽 1.4px。

**右上角图例**：白色框 + 彩色虚线 + 标签 L1/L2/L3。

**底部振幅曲线**：三条纵线位置的振幅-样点曲线，叠加在同一坐标系中。

**意义**：指数位是 Stage4 算法的**核心压缩对象**。指数位图呈现明显的**块状结构**（相邻区域颜色相近），说明相邻采样点的指数值高度相关——这正是 CNN 可以建模的"空间相干性"。

**与算法的联系**：
- 指数位图上的**颜色块**对应指数值的空间聚类
- CNN 从因果邻域预测当前点的指数值概率分布
- 块状结构越明显 → CNN 预测越准 → Range Coder 编码越短 → 压缩率越高
- 三条纵线用于定量分析：比较不同道位置的振幅变化，验证指数预测的一致性

---

### 5.4 尾数位分布（Mantissa Field）

**数据源**：`mantissa.high_byte_grid` — 尾数高字节（bit 16-22）。每个 float32 的 23-bit 尾数右移 16 位，得到 0-127 的 7-bit 值。

**渲染**：使用 Viridis 色阶（与指数位相同配色风格）。

**意义**：尾数位是 float32 中最接近**均匀随机噪声**的成分。尾数低位（bit 0-7）的熵接近 1.0 bit/bit，几乎无法压缩。

**数据分析洞察**：
- 尾数高位（bit 16-22）有微弱的模式（可能反映数据的精度需求）
- 尾数低位（bit 0-7）完全随机
- Stage4 对尾数位使用 zlib 压缩，虽然压缩率不高（约 10-20%），但无损

## 6. 尾数位平面视图

点击局部位统计面板标题栏的 **[尾数位]** 按钮切换到尾数位平面视图。

### 6.1 视图布局

```
┌───────────────────┬──────┬───────────────────┬────────┐
│    高 8 位         │ fold │    低 8 位         │  图例   │
│    (m22–m15)      │m14:m8│    (m7–m0)        │        │
│                   │      │                   │        │
│ 40浮点值×8列       │      │ 40浮点值×8列       │ 0      │
│ 每格=单个bit      │      │ 每格=单个bit       │ other 1│
│                   │      │                   │managed1│
└───────────────────┴──────┴───────────────────┴────────┘
```

### 6.2 数据提取

取选区内**中间道的 40 个连续浮点值**，逐 bit 提取并着色：

```javascript
for (sample = 0; sample < 40; sample++) {
    var mantissa = fullData[middleTrace][sample];  // uint32, 23-bit mantissa
    for (bit = 22; bit >= 15; bit--) {            // 高 8 位
        var val = (mantissa >> bit) & 1;
        var isManaged = (bit >= 19 && bit <= 22);  // managed nibble m22-m19
        color = val ? (isManaged ? '#FCD69A' : '#E5E5E5') : '#FFFFFF';
        drawCell(trace, sample);
    }
}
// 低 8 位同理，managed nibble = m3-m0
```

### 6.3 着色方案

仿 `find_postdata_motivation_rois.py` 的 `MANTISSA_BIT_CMAP`：

| 颜色 | 十六进制 | 含义 |
|------|---------|------|
| ⬜ 白色 | `#FFFFFF` | bit = 0 |
| ⬜ 浅灰 | `#E5E5E5` | bit = 1（普通位） |
| 🟡 琥珀 | `#FCD69A` | bit = 1（managed nibble：高位 m22-m19 / 低位 m3-m0） |

### 6.4 什么是 Managed Nibble？

**Managed Nibble** 是 PathL-lite 算法中的概念——对尾数的**高位 4 bits（m22-m19）**和**低位 4 bits（m3-m0）**进行特殊管理（如 RLE 编码），对中间的 m18-m4 不加干预。

位平面视图中，managed nibble 的 bit=1 用琥珀色 `#FCD69A` 标记，区别于普通位的浅灰 `#E5E5E5`。这样用户可以直观看到：

- **高位 m22-m19**：往往呈现一定的模式（不是完全随机的 0/1 交替）
- **低位 m3-m0**：接近均匀随机
- **中位 m18-m4**（fold 隐藏）：介于两者之间

**与算法的联系**：Stage4 的辅助压缩策略（PathL-lite 等）正是利用了尾数高低位的这种特性差异——高位有微弱结构可被管理，低位纯随机只能用熵编码。

## 7. 指数位信息与 Stage4 算法的关联

这是整个局部分析功能**最核心的学术价值**——通过可视化验证 Stage4 算法的理论前提。

### 7.1 问题：为什么只压缩指数位？

Stage4 的核心设计决策是**三通道分离**：只对中间的 8 个指数位做 CNN 预测压缩，对符号位和尾数位使用传统方法。

这个决策基于以下观察（局部分析功能可视化验证）：

| 位段 | 统计特性 | 可视化表现 | 压缩策略 |
|------|---------|-----------|---------|
| 符号位 (1 bit) | 接近均匀二值分布，无空间相关性 | 蓝/白像素随机散布 | zlib 通用压缩 |
| 指数位 (8 bits) | **高度空间相干**，相邻值相关 | 明显的色块/聚类 | **CNN 预测 + Range Coder** |
| 尾数位 (23 bits) | 高位有弱模式，低位接近噪声 | 热力图无明显结构 | zlib / PathL-lite |

### 7.2 指数位的空间相干性——来自局部分析的证据

在「指数位分布」渲染图中，清晰可见：

1. **色块聚类**：相邻 pixel 的颜色相近（如一大片绿色区域），说明相邻采样点的指数值接近——这是 CNN 可以利用的空间相关信息。

2. **振幅曲线联动**：拖拽纵线后，振幅曲线呈现连续的波形而非随机抖动。三条线在同一区域的变化趋势相似——进一步证明指数值（决定振幅的量级）在空间上是平滑变化的。

3. **与灰度图对照**：在灰度图的同一位置，振幅变化是渐进的。指数值作为"振幅的量级指示器"，自然也具有渐进性。

### 7.3 因果 CNN 如何利用这种相干性

```
灰度图上的一个点 (profile=p, trace=t, sample=s)
                              ↓
     在 profile-trace 平面上提取 17×17 因果 Patch
                              ↓
     仅填入已编码的因果区域（对角线左侧+上方）
                              ↓
     Small2DCNN 接收 Patch → 输出 256 类概率
                              ↓
     预测: "指数值 == X" 的概率分布
```

因为指数值在空间中缓慢变化，CNN 通过观察因果邻域的指数值，就能较准确地预测当前点的指数值。局部分析中的**纵线拖拽**功能直观展示了这种空间一致性——在相邻道之间，振幅曲线高度相似。

### 7.4 Range Coder 如何将预测转化为压缩

```
CNN 输出: P(指数=128) = 0.85
                  ↓
    CDF 中 Symbol 128 占据 85% 的区间宽度
                  ↓
    Range 编码: 区间从 [0, 1) 缩窄到 [cdf[128], cdf[129]) ≈ [0.10, 0.95)
                  ↓
    所需 bit 数 = -log₂(0.85) ≈ 0.23 bits ← 仅需 0.23 bits 编码 8-bit 指数！
```

**关键洞察**：如果 CNN 预测完全准确（概率 = 1.0），编码该指数只需 **0 bit**。局部分析功能通过指数位图和振幅曲线，定性验证了"指数值可预测"这一核心假设——正是这一假设使 2-3 倍的压缩比成为可能。

### 7.5 尾数位：为什么无法用 CNN 压缩

在「尾数位平面」视图中，bit 网格呈现均匀的 0/1 交替，无明显模式：

- 相邻 bit 之间的 0/1 分布接近 50/50
- 同一 bit 位置的相邻采样点之间也无明显相关性
- 高位（m22-m15）有微弱偏差（managed nibble 区域琥珀色略多），但远不足以支撑 CNN 建模

这解释了 Stage4 对尾数位采用 zlib 直接压缩的原因——CNN 对纯随机数据无效，通用压缩器反而更合适。

### 7.6 局部分析的学术价值总结

| 问题 | 局部分析如何回答 |
|------|----------------|
| 为什么要分离指数位？ | 指数位图显示色块聚类 → 有空间结构；符号位图随机散布 → 无结构 |
| CNN 为什么能预测指数？ | 振幅曲线在相邻道间高度相似 → 因果邻域携带预测信息 |
| 为什么不对尾数用 CNN？ | 尾数位平面网格接近均匀随机 → CNN 无法建模 |
| 压缩比的来源是什么？ | 指数位概率分布高度集中（高概率 bin 占绝大多数）→ 熵远低于 8 bits |

```json
        "high_byte_grid": [[...], ...],     // 尾数高字节矩阵
        "full_grid": [[...], ...]           // 完整尾数值 (uint32)
    }
}
```

### 8.4 实现细节

- 使用 `struct.unpack('>f', ...)` 读取 IEEE 浮点
- 自动识别 IBM 浮点格式（format_code=1）并转换
- 使用 numpy 向量化计算位统计（`np.bincount`, `np.sum`）
- 指数熵值使用 `-Σ(p × log₂(p))` 公式
