## 1. 算法总览

### 1.1 一句话描述

**Stage4** 是一种基于卷积神经网络的无损浮点数据压缩算法，通过将 Float32 分解为 Sign / Exponent / Mantissa 三个位段，仅对空间可预测的指数段进行 CNN 建模压缩，其余段使用传统熵编码。

### 1.2 压缩流程概览

```
输入: 三维 Float32 数据体 (Profile × Trace × Sample)
                             ↓
                  ┌──────────┴──────────┐
                  │   IEEE 754 位分解    │
                  │ u32 = float.view(uint32) │
                  └──────────┬──────────┘
                             ↓
         ┌───────────────────┼───────────────────┐
         ↓                   ↓                   ↓
    ┌─────────┐        ┌──────────┐        ┌──────────┐
    │ 符号位   │        │  指数位   │        │  尾数位   │
    │ 1 bit    │        │  8 bits   │        │ 23 bits   │
    │ 高度随机  │        │ 空间相干  │        │ 接近噪声  │
    └────┬────┘        └────┬─────┘        └────┬─────┘
         ↓                   ↓                   ↓
    zlib 压缩          ┌─────┴─────┐         zlib 压缩
                       │ 因果特征   │
                       │ CNN 预测   │
                       │ Range 编码 │
                       └─────┬─────┘
         ↓                   ↓                   ↓
    ┌─────────┐        ┌──────────┐        ┌──────────┐
    │sign.zlib│        │  .s4rc   │        │mant.zlib │
    │  ~12KB  │        │  ~85MB   │        │  ~37MB   │
    └────┬────┘        └────┬─────┘        └────┬─────┘
         └──────────────────┼──────────────────┘
                            ↓
                    总比特流 = s4rc + sign + mant
                    压缩比 ≈ 原始 / 总比特流
```

### 1.3 核心技术栈

| 环节 | 技术 | 说明 |
|------|------|------|
| 位分解 | `numpy.view(uint32)` | 零拷贝类型转换，瞬间完成 |
| 特征提取 | 因果 Patch 构建 | 仅使用已编码邻域，保证编解码一致性 |
| 概率预测 | Small2DCNN (PyTorch) | 3 层卷积，~5000 参数，推理极快 |
| 熵编码 | Range Coder (Numba JIT) | 整数算术编码，接近理论熵界 |
| 辅助压缩 | zlib | 对符号和尾数的通用压缩 |

## 2. 数据流全景

### 2.1 输入数据形态

```
三维地震数据体:
  Profile (剖面)  ×  Trace (道)  ×  Sample (采样点)
      350         ×     600      ×      2001

每个元素: float32 (4 字节)
总大小: 350 × 600 × 2001 × 4 = 1,680,840,000 字节 ≈ 1.6 GB
```

### 2.2 编码器数据流（详细）

```
┌──────────────────────────────────────────────────────────┐
│                      编码器 (Encoder)                      │
│                                                          │
│  Float32 数据体 [1.6GB]                                   │
│      ↓                                                   │
│  u32 = data.view(np.uint32)   ← 零拷贝类型转换             │
│      ↓                                                   │
│  signs = (u32 >> 31) & 0x1        → sign.zlib [12KB]     │
│  exps  = (u32 >> 23) & 0xFF      → ↓                    │
│  mants = u32 & 0x7FFFFF          → mant.zlib [37MB]     │
│                                      ↓                    │
│      ┌───────────────────────────────────────────────┐   │
│      │         Stage4 指数压缩管线                      │   │
│      │                                                │   │
│      │  for each diagonal = 0 to P+T+S-3:             │   │
│      │    for each (p,t,s) where p+t+s == diagonal:   │   │
│      │      ┌───────────────────────────────────────┐ │   │
│      │      │ ① 构建因果 Patch (17×17×C)            │ │   │
│      │      │   仅使用已编码坐标的指数值              │ │   │
│      │      │   未编码区域填充 0                      │ │   │
│      │      └──────────────┬────────────────────────┘ │   │
│      │                     ↓                          │   │
│      │      ┌───────────────────────────────────────┐ │   │
│      │      │ ② Small2DCNN 推理                      │ │   │
│      │      │   Patch → 3×Conv2D → GAP → FC256      │ │   │
│      │      │   输出: 256 类 softmax 概率分布         │ │   │
│      │      └──────────────┬────────────────────────┘ │   │
│      │                     ↓                          │   │
│      │      ┌───────────────────────────────────────┐ │   │
│      │      │ ③ 概率→CDF 量化                        │ │   │
│      │      │   probs × 32768 → floor → 余量分配     │ │   │
│      │      │   输出: cdf[0..256] 累积分布数组        │ │   │
│      │      └──────────────┬────────────────────────┘ │   │
│      │                     ↓                          │   │
│      │      ┌───────────────────────────────────────┐ │   │
│      │      │ ④ Range Encoder                        │ │   │
│      │      │   range = high - low + 1               │ │   │
│      │      │   low  += range × cdf[symbol] / total  │ │   │
│      │      │   high  = low + range × prob / total   │ │   │
│      │      │   输出: 比特流追加                      │ │   │
│      │      └───────────────────────────────────────┘ │   │
│      └───────────────────────────────────────────────┘   │
│                          ↓                               │
│                    .s4rc 文件 [85MB]                      │
└──────────────────────────────────────────────────────────┘

总输出: compressed.s4rc + sign.zlib + mant.zlib ≈ 122MB
压缩比: 1600MB / 122MB ≈ 13.1x (仅指数) → 综合约 2.96x
```

### 2.3 解码器数据流

解码器是编码器的**镜像**——按相同的对角线顺序，使用 Range Decoder 从比特流中读取符号，用完全相同的 CNN 预测概率分布来解码：

```
.s4rc [85MB] + sign.zlib [12KB] + mant.zlib [37MB]
        ↓
Range Decoder 读比特流 → 按对角线顺序解码每个符号
        ↓                           ↓
   Small2DCNN (相同模型)         因果 Patch 构建
   ↓ 预测概率分布                  (相同逻辑)
   ↓ 解码当前符号
        ↓
恢复指数值 → 解压符号/尾数 → 重建 Float32 → 验证无损
```

## 3. 步骤 1：Float32 位拆解

### 3.1 做了什么

将每个 32-bit 浮点数按 IEEE 754 标准拆为三个独立的位段。

### 3.2 代码

```python
# algorithm/common.py
def extract_float_components(data_float32):
    # 零拷贝：将 float32 内存直接解释为 uint32
    data_u32 = np.asarray(data_float32, dtype=np.float32).view(np.uint32)

    # 符号位：bit 31，右移 31 位取最低位
    signs = ((data_u32 >> 31) & 0x1).astype(np.uint8)

    # 指数位：bit 30-23，右移 23 位取低 8 位
    exps = ((data_u32 >> 23) & 0xFF).astype(np.uint8)

    # 尾数位：bit 22-0，取低 23 位
    mants = (data_u32 & 0x7FFFFF).astype(np.uint32)

    return signs, exps, mants
```

### 3.3 每个位的含义

```
┌──────┬────────────────┬──────────────────────────┐
│ Sign │   Exponent     │       Mantissa           │
│ 1 bit│    8 bits      │       23 bits            │
│ Bit31│   Bit30-Bit23  │      Bit22-Bit0          │
├──────┼────────────────┼──────────────────────────┤
│ 0=正  │ biased exponent│  fraction (隐含 1.xxx)    │
│ 1=负  │ 实际指数 = e-127 │                        │
└──────┴────────────────┴──────────────────────────┘

Float32 值 = (-1)^sign × 2^(exp-127) × (1 + mant/2^23)

例: 0|10000001|01000000000000000000000
    = (-1)^0 × 2^(129-127) × (1 + 0.25)
    = 1 × 4 × 1.25
    = 5.0
```

### 3.4 为什么要拆开

| 位段 | 统计特性 | 处理策略 |
|------|---------|---------|
| 符号（1 bit） | 约 50% 概率为 0 或 1，无空间相关性 | zlib 直接压缩 |
| 指数（8 bits） | **空间相干**——相邻点振幅相近，指数值平滑变化 | CNN 预测建模 |
| 尾数（23 bits） | 高位有微弱模式，低位纯随机 | zlib 压缩 |

**关键洞察**：Float32 的三个位段在统计性质上截然不同——强行统一压缩（如 zstd）会抹杀指数段的可预测性，而用 CNN 建模符号或尾数又徒劳无功。分离处理是 Stage4 的第一个核心设计决策。

### 3.5 在算法演示中的呈现

**3D 演示（Step 1）**：32 个彩色立方体沿 X 轴排列，红色=符号，橙色=指数，青色=尾数。亮的立方体=bit=1，暗的=bit=0。用户可直观看到当前采样点的 32 位表示。

**2D 演示（Step 1）**：同样 32 个像素方块，带发光阴影效果，分组标注 Sign/Exponent/Mantissa。

## 4. 步骤 2：因果特征提取

### 4.1 做了什么

对每个待编码的采样点，从其空间邻域提取一个 17×17 的 2D Patch 作为 CNN 输入。**仅使用已经编码过的位置**（因果区域），未编码位置的指数值用 0 填充。

### 4.2 因果约束

```
Profile-Trace 平面上的对角线扫描顺序：

  t→  0  1  2  3  4  5  6
p ↓  ┌───┬───┬───┬───┬───┬───┬───┐
 0   │▓▓▓│▓▓▓│▓▓▓│▓▓▓│▓▓▓│▓▓▓│░░░│   ▓ = 已编码 (Causal)
     ├───┼───┼───┼───┼───┼───┼───┤   ■ = 当前点
 1   │▓▓▓│▓▓▓│▓▓▓│▓▓▓│▓▓▓│░░░│░░░│   ░ = 未编码 (Future)
     ├───┼───┼───┼───┼───┼───┼───┤
 2   │▓▓▓│▓▓▓│▓▓▓│▓▓▓│░░░│░░░│░░░│
     ├───┼───┼───┼───┼───┼───┼───┤   编码路径: 对角线方向
 3   │▓▓▓│▓▓▓│▓▓▓│▓■▓│░░░│░░░│░░░│   diag = p + t + s 递增
     ├───┼───┼───┼───┼───┼───┼───┤
 ...                                  当前点 ■: 只能看到 ▓ 区域
                                      不可看到 ░ 区域
```

### 4.3 编码顺序

```
for diag in 0 ... (P + T + S - 3):
    for each (p, t, s) where p + t + s == diag:
        encode(p, t, s)
```

对角线顺序保证：当前点的所有"上游"（p' + t' + s' < p + t + s）邻居都已被编码。

### 4.4 17×17 Patch 构建

在 profile-trace 平面上，以当前点 (p, t, s) 为中心，取 17×17 的邻域：

```
┌─────────────────┐
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │  0-7 行: 因果区域（填入真实指数值）
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓░░░  │  第 8 行: 因果部分 + 当前点 + 未来部分
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓░░░  │
│ ...               │
└─────────────────┘
  ■ = 当前点 (第 8 行第 8 列)
  ▓ = 因果可用（真实值）
  ░ = 因果不可用（填 0）
```

在 `diagonal_causal_edge` 模式下，特征通道包括：
- 通道 1：指数值（因果区域=真实值，未来=0）
- 通道 2：可用掩码（因果=1，未来=0）
- 通道 3：因果掩码
- 通道 4+：残差值、预测值等

### 4.5 为什么要保证因果

**解码器必须能复现编码器的预测**。如果 CNN 输入包含了未编码位置的信息，编码器可以用这些信息做更准的预测——但解码器在解码时尚未获得那些位置的数据，无法复现同样的预测，导致解码失败。

因果约束保证了编码器和解码器**始终看到完全相同的信息**，这是 Stage4 实现无损压缩的数学基础。

### 4.6 在算法演示中的呈现

**3D 演示（Step 2）**：10×10×10 体素网格旋转展示，对角扫描线推进，青色=已编码，粉色=当前，灰色=未编码。右侧浮动 17×17 Patch 平面，粉色中心格 + 青色因果区 + 灰色非因果区。

## 5. 步骤 3：CNN 概率预测

### 5.1 做了什么

将因果 Patch 送入 Small2DCNN，输出 256 类概率分布，表示"下一个指数值各有多少概率"。

### 5.2 网络结构

```
输入: 17×17×C 因果 Patch
  ↓
┌──────────────────────┐
│ Conv2D 3×3, C→16     │  ← 提取局部空间特征
│ + ReLU               │
├──────────────────────┤
│ Conv2D 3×3, 16→16    │  ← 高层特征组合
│ + ReLU               │
├──────────────────────┤
│ Conv2D 3×3, 16→16    │  ← 进一步抽象
│ + ReLU               │
├──────────────────────┤
│ AdaptiveAvgPool2d(1) │  ← 全局空间池化
│ Flatten              │
├──────────────────────┤
│ Linear(16→256)       │  ← 分类头，输出 256 类
└──────────────────────┘
  ↓
Softmax → 256 维概率向量 (p0, p1, ..., p255)
```

**网络特点**：
- 仅 3 层卷积，约 5000 个参数——极轻量
- 输入通道数 C 取决于特征模式（`diagonal_causal_edge` 约 2-4 通道）
- 在 GPU 上单个采样点推理约 0.1μs

### 5.3 目标模式

| 模式 | 符号含义 | 说明 |
|------|---------|------|
| `raw` | 指数本身 (0-255) | 直接预测指数值 |
| `residual` | 指数 - LOCO-I 预测值 | 预测残差——分布更集中，压缩更高效 |

**LOCO-I 预测器**（residual 模式）：

```python
def predictor_loco_i_2d(plane, t, s):
    # 边界情况: 返回 0 或邻居值
    if t == 0 and s == 0: return 0        # 左上角
    if t == 0: return plane[0, s-1]      # 第一行
    if s == 0: return plane[t-1, 0]      # 第一列
    # 核心公式: clamp(left + up - up_left, 0, 255)
    return clamp(plane[t, s-1] + plane[t-1, s] - plane[t-1, s-1], 0, 255)

# 残差符号
symbol = (real_exponent - LOCO_I_prediction) mod 256
```

残差模式利用了指数值的空间连续性——LOCO-I 预测通常很准，残差集中在 0 附近，分布更尖锐，熵更低，压缩效率更高。

### 5.4 为什么用 256 类而不是连续回归

指数值是 0-255 的离散整数（8 bits）。使用分类（256 类 softmax）而非回归的好处：

1. **精确概率**：分类输出每个值的精确概率，可直接用于算术编码
2. **避免量化**：回归需要将连续输出再量化，引入额外误差
3. **端到端优化**：交叉熵损失直接最小化编码期望长度

### 5.5 在算法演示中的呈现

**3D 演示（Step 3）**：256 根概率柱沿弧形排列，柱高∝概率，彩虹色阶。粉色 Beacon 光柱标记真实 Symbol 位置。CNN 结构以层叠方块模型展示。

## 6. 步骤 4：Range 算术编码

### 6.1 做了什么

将 CNN 输出的 256 类概率分布量化为 CDF（累积分布函数），用 Range Encoder 将真实符号编码为比特流。

### 6.2 概率 → CDF 量化

```python
# algorithm/codec.py
def probs_to_cdf(probs, total_freq=32768):
    # 1) 缩放: probs × 32768
    freqs = floor(probs * total_freq)

    # 2) 分配余量: 总和不足 total_freq 的差额分配给小数部分最大的 bin
    remainder = total_freq - sum(freqs)
    top_indices = argsort(-fractional_parts)[:remainder]
    for i in top_indices: freqs[i] += 1

    # 3) 确保非零 bin 至少 1 计数
    for zero_bin with prob>0: 从最高 bin 借 1

    # 4) 累积求和
    cdf[0] = 0
    cdf[1:] = cumsum(freqs)  # cdf[i] = Σ freq[0..i-1]
    cdf[256] = total_freq
    return cdf  # shape: (257,)
```

### 6.3 Range 编码过程

```python
# 伪代码
encoder = RangeEncoder(num_state_bits=32)

# high = 2^32 - 1, low = 0
# 对每个符号 symbol:
for each symbol at (p, t, s):
    feature = build_causal_patch(p, t, s)
    probs = CNN(feature)
    cdf = probs_to_cdf(probs, 32768)

    # Range 编码核心公式
    range = high - low + 1
    low  = low  + range * cdf[symbol] / 32768       # 区间下界缩窄
    high = low  + range * prob[symbol] / 32768 - 1   # 区间上界缩窄

    # 当 high 和 low 的高位相同时，输出确定 bit
    while ((low ^ high) & 0x80000000) == 0:
        output_bit(low >> 31)
        low = (low << 1) & 0xFFFFFFFF
        high = ((high << 1) & 0xFFFFFFFF) | 1

encoder.finish()  # 输出剩余 bits
```

### 6.4 直观理解

```
初始区间: [0.000000, 1.000000)    ←  32-bit 整数等价于 [0, 2^32-1]
                                                     ↓
symbol=128, prob=0.85, cdf[128]=0.10
    区间缩窄到: [0.10, 0.95)     ←  prob 越高，区间越宽
                                                     ↓
下一个 symbol 再次缩窄...
    [0.10, 0.95) → [0.12, 0.38)
                                                     ↓
当 high 和 low 的 MSB 相同时 → 输出 1 bit
当区间太窄时 → 重归一化（左移）
                                                     ↓
最终输出: 累计输出约 0.23 bits/symbol (而非 8 bits/symbol)
```

### 6.5 为什么用算术编码而不是 Huffman

| 编码方式 | 效率 | 说明 |
|---------|------|------|
| Huffman | 亚优 | 每个符号需要整数个 bit（≥1），概率 0.85 的符号仍需至少 1 bit |
| Arithmetic/Range | **接近熵界** | 可以为单符号分配 < 1 bit，-log₂(0.85)=0.23 bits |

**Range Coder 是 Stage4 实现高压缩率的关键**——指数位的理论熵（经过 CNN 预测后）通常为 2-4 bits，远低于原始的 8 bits。只有算术编码才能接近这一理论下界。

### 6.6 在算法演示中的呈现

**3D 演示（Step 4）**：主概率条（256 段彩色）→ 粉色箭头指向当前 Symbol → 三层递归隧道模拟区间缩窄 → 底部 bitstream 方块（绿=1, 灰=0）。

**2D 演示（Step 4）**：概率条 + 放大视图 + bitstream + 真实编码区间和累计 bit 数。

## 7. 步骤 5：符号与尾数辅助压缩

### 7.1 做了什么

符号位和尾数位不使用 CNN 建模，而是用 zlib（或 zstd、lzma、RLE）直接压缩。

### 7.2 符号位压缩

```python
# 1 bit/sample → packbits 为 uint8 → zlib 压缩
signs = ((u32 >> 31) & 0x1).astype(np.uint8)
packed = np.packbits(signs.flatten())
compressed = zlib.compress(packed.tobytes(), level=1)
```

**为什么不压缩**：每个 bit 约 50% 概率为 0/1，无空间相关性，分布接近随机。CNN 对其无效，zlib 已足够。

### 7.3 尾数位压缩

```python
# 23 bits/sample → 提取为 uint8 数组 → zlib 压缩
mants = (u32 & 0x7FFFFF).astype(np.uint32)
mant_bytes = np.zeros((mants.size, 3), dtype=np.uint8)
mant_bytes[:, 0] = (mants & 0xFF).astype(np.uint8)
mant_bytes[:, 1] = ((mants >> 8) & 0xFF).astype(np.uint8)
mant_bytes[:, 2] = ((mants >> 16) & 0xFF).astype(np.uint8)
compressed = zlib.compress(mant_bytes.tobytes(), level=1)
```

**尾数位特性**：高位（bit 16-22）有微弱偏差（非均匀 0/1），低位（bit 0-7）接近完全随机。整体熵约 20-22 bits，zlib 可压缩约 10-20%。

### 7.4 在算法演示中的呈现

**3D 演示（Step 5 甜甜圈图）**：三色弧段（橙=Stage4 指数, 红=符号, 青=尾数）直观展示三通道在总压缩后大小中的占比。

## 8. 步骤 6：三通道合并与解压

### 8.1 编码输出

压缩产出 **3 个文件**（+ 1 个头文件）：

| 文件 | 大小示例 | 内容 |
|------|---------|------|
| `compressed.s4rc` | ~85 MB | 指数位 Stage4 CNN+Range 编码 |
| `sign.zlib` | ~12 KB | 符号位 zlib 压缩 |
| `mant.zlib` | ~37 MB | 尾数位 zlib 压缩 |
| `sgy_headers.json` | ~5 KB | SGY 头文件（供重建用） |

压缩比：`363 MB / (85+0.01+37) MB ≈ 2.96`

### 8.2 解码（解压）

解码是编码的**完整逆过程**：

```
.s4rc + sign.zlib + mant.zlib
        ↓
┌───────────────────────────────────┐
│ Range Decoder 按对角线顺序逐点解码  │
│   ↓                               │
│ 对每个 (p,t,s):                    │
│   ├ 构建因果 Patch (与编码完全相同)  │
│   ├ CNN 预测概率分布 (与编码完全相同) │
│   ├ CDF 量化                       │
│   └ Decoder.decode(cdf) → symbol  │
│   ↓                               │
│ 恢复指数值 (residual 模式需逆 LOCO-I) │
│   + 解压符号位 + 解压尾数位         │
│   ↓                               │
│ 重建 Float32: sign | exp<<23 | mant│
└───────────────────────────────────┘
        ↓
逐元素比对原始数据 → MSE=0 → 无损验证通过
```

### 8.3 无损验证

解压后逐元素比对：

```
原始: [3.14, -123.45, 0.0, ...]
解压: [3.14, -123.45, 0.0, ...]
差异: [0, 0, 0, ...]
→ SHA256 校验通过
→ 无损压缩确认
```

## 9. 算法为什么这样设计

### 9.1 为什么分离三个位段

| 设计理由 | 说明 |
|---------|------|
| **指数位可预测** | 空间相关性 → CNN 可以建模 → 熵从 8 bit 降到 2-4 bit |
| **符号位不可预测** | 纯随机 0/1 → CNN 无能为力 → 直接 zlib |
| **尾数位半随机** | 高位有微弱偏差但远不足以支撑 CNN 建模 → zlib 或 PathL-lite |
| **分离最大化压缩效率** | 每个位段用最适合的方法，总体压缩最优 |

### 9.2 为什么用对角线编码顺序

```
对角线扫描:
  p+t+s = 0, 1, 2, ... 递增

好处:
  ① 保证因果性——当前点的所有上游都已编码
  ② 编解码对称——解码器按相同顺序，看到相同上下文
  ③ 空间局部性——相邻的 (p,t,s) 在编码顺序中也接近
  ④ 并行友好——同一对角线的点相互独立，可批量推理
```

### 9.3 为什么用轻量 CNN 而不是大模型

```
Small2DCNN:
  - 3 层卷积, ~5000 参数
  - 推理: 0.1 μs/sample (GPU), ~10 μs/sample (CPU)
  - 训练: 几小时收敛

大模型（如 ResNet）:
  - 数千万参数
  - 推理: 远慢于 Range Coder 编码速度
  - 过拟合风险

权衡:
  压缩率提升 vs 编码速度下降。
  Stage4 选择轻量模型保证编码吞吐量。
```

### 9.4 为什么用 Range Coding 而不是 Huffman

```
例: 符号概率为 0.85
  Huffman:                 需要 ≥ 1 bit
  Range Coding: -log₂(0.85) ≈ 0.23 bit
  → 效率差 ~4×
```

算术编码可为高概率符号分配 < 1 bit 的码字，是接近理论熵界的唯一手段。
